## Why M2 next

M0 (delta loop) and M1 (processes) are done; the M3 LT-TLM slice is done — **but the
implementation jumped from M1 to a partial M3, skipping M2**. M2 is the load-bearing
foundation the rest of the design assumes:

- Sockets are *port + export pairs* (§6d); today's `systemrs-tlm2` socket registry is a
  bespoke shortcut that bypasses the generic `Interface`/`Port`/`Export` two-phase binding.
- Hierarchical TLM platforms, multi/passthrough sockets, and AT topology (M4) need
  hierarchical binding + the object hierarchy.
- Twin **introspection/tracing** (M5) keys off hierarchical object names.
- The four `Elaborate` lifecycle callbacks exist but **are never driven** — there is no
  elaboration barrier at all yet.

## Design approach (one line)

M2 is a **strictly additive overlay** on the green M0/M1/M3 core that **does not touch the
scheduler hot path** (`crunch`/`trigger`/`update`). A 4th `Copy` generational id `ObjectId`
backs an `ObjectStore` (arena + name table + scope stack + four per-bucket elaborator
registries) registered as a `Sim` service — exactly like the existing `SocketRegistry`. The
elaboration driver is invoked from `Sim::run_until` via a dependency-inverted hook in `Inner`
so the kernel (L1) never names a core (L2) type.

## Milestone 2 exit criteria (from §12)

1. **EC1** — a two-level hierarchy elaborates with correct **unique dot-joined names**.
2. **EC2** — a **channel bind** resolves via two-phase bind + `complete_binding`.
3. **EC3** — a **hierarchical port-to-port** bind resolves (depth-first parent flatten).
4. **EC4** — **binding after `start` is a compile error _or_ a clean `Err`**.
5. **EC5** — the **construction-done fixpoint** converges (a module created in
   `before_end_of_elaboration` still gets its callback).
6. **EC6** — the **four lifecycle callbacks** fire in fixed registry order; `end_of_simulation`
   fires exactly once.
7. **EC7** — **port-policy cardinality** is enforced at the end of `complete_binding`.

3. Phase C — the driver and its wiring (the only schedule-affecting change, isolated and gated): M2-06 (per-bucket fixpoint + 4 callbacks in fixed bucket order + complete_binding orchestration WITH the borrow-release discipline) depends on M2-03 and M2-05. Immediately after M2-07, run an incremental build-verification sweep AND re-run the existing integration suite — the highest-value regression gate (confirms a no-op elaboration is bit-identical, and end_of_simulation does not multi-fire).

4. Phase D — front door + ergonomics: M2-08 (cx.module/ScopeGuard/Builder<M> with Send-aware m.thread) needs M2-04. 5. Phase E — reconcile sockets last so it re-targets a PROVEN binding API rather than co-evolving with it: M2-09 refactors tlm2 sockets onto a FORWARD-ONLY Port/Export pair (needs M2-05), keeping the closure registry as resolved storage and bind/b_transport as thin wrappers, with the bw scaffold DROPPED to M4. 6. Phase F — prove + verify: M2-13 (two-level hierarchy example + test covering all seven exit criteria, the golden delta-accounting test, plus empirical re-verification of every existing timing assertion) then M2-14 (facade re-exports + the FINAL build-verification consolidation fmt→clippy -D warnings→test→build --release→doc→deny→audit, including license/advisory clearance for the new proc-macro deps). 7. Rationale: foundation (A) and binding (B) are pure additions with their own unit tests and zero risk to existing behavior; the single schedule-affecting change is concentrated in M2-07 and immediately gated by an incremental sweep + integration re-run (no longer deferred entirely to M2-14, per the critic); the socket reconcile (E) and macro (D) consume a proven API; the fidelity-critical pieces (per-bucket fixpoint M2-06, two-phase bind + insert_parent flatten M2-05) are ported against the vendored C++ with targeted N=1/N>1/2-deep AND re-entrancy tests. ## Risks & mitigations

- **Risk:** END_OF_SIMULATION fired from run_until's starvation exit multi-fires across stepped run_until calls and at the wrong moment (mid-simulation), under-satisfying exit criterion 6 (critic's primary gap).
  - **Mitigation:** Do NOT fire end_of_simulation from run_until. Add an explicit Sim::end_of_sim() guarded by an `ended: bool` latch so it fires EXACTLY ONCE; the flat path calls it explicitly (or via Sim::Drop as a backstop), and Kernel<Running>::finish()/Drop calls it at the correct single teardown moment. A dedicated test asserts exactly-once across multiple run_until calls + finish().
- **Risk:** A single construction-done cursor over a mixed elaborators Vec cannot reproduce SystemC's four independent per-registry cursors advancing at different rates — a concrete fidelity bug for the fixpoint (EC5/EC6) (critic's fidelity finding).
  - **Mitigation:** Replace the single cursor with FOUR independent ElabBuckets (ports/exports/prim_channels/modules), each with its own construction_done_cursor, mirroring sc_port/sc_export/sc_prim_channel/sc_module registries. object_kind() routes each elaborator into the correct bucket at registration. A mixed-kind fixpoint test (a callback creating both a port and a submodule) pins the per-bucket behavior.
- **Risk:** RefCell double-borrow PANIC: the driver borrows the ObjectStore to iterate elaborators while a before_end_of_elaboration body calls cx.module, re-entering the SAME store to insert/push_scope — a second borrow_mut while the iteration borrow is live. This is the single highest-probability runtime panic (critic's central hazard).
  - **Mitigation:** Borrow-release discipline (the arena+id design enables it): the driver reads (ObjectId, Rc<RefCell<dyn Elaborate>>) under a brief store borrow, clones the Rc OUT, DROPS the store borrow, THEN calls the callback — so the re-entrant cx.module sees no live store borrow. This mirrors the proven socket-registry pattern (socket.rs clones the callback Rc out before calling). Because everything is referenced BY ObjectId (not &T), the driver re-borrows the store by id after each callback to re-read the bucket cursor/len. A re-entrancy test (cx.module from inside a callback) proves no panic.
- **Risk:** complete_binding's recursive insert_parent reads the parent's resolved Bound vec while mutating the child's Recorded vec, both behind RefCell and the ObjectStore RefCell — aliasing the borrow across the recursion (C++ has no borrow checker; a naive line-by-line port panics) (critic's borrow finding).
  - **Mitigation:** Thread ObjectIds and re-borrow per access, never holding &mut BindState or &mut ObjectStore across the recursive call: take/clone the child's BindElems OUT, drop the borrow; look up the parent by id, clone its PortMeta Rc out, drop the store borrow, recurse; then re-borrow the parent's now-Bound vec by id. A 2-deep port-to-port chain test proves no double-borrow.
- **Risk:** Inserting the init pass shifts the observable schedule by a delta for any model that wrote a channel during elaboration; O5 (commit-only vs full-notify) was decided silently and the schedule-invariance claim was asserted-not-proven (critic's correctness finding).
  - **Mitigation:** run_initial_commit replays the EXISTING UPDATE block once (commit values + delta-notify), with an EARLY RETURN when update_queue is empty so no-write models (counter/rv32i) are bit-identical (no change_stamp/delta_count bump). The O5 decision (commit values + queue value-changed for the first delta) is PINNED by a golden test asserting delta_count/change_stamp/now against hand-computed values for a write-during-elaboration model. M2-07 re-runs the full integration suite immediately.
- **Risk:** The structural bw-socket scaffold (Port<BwTransport>/Export, crossed double-bind) recorded but unresolved would trip OneOrMore cardinality enforcement at complete_binding on every working LT socket (rv32i), and pulls M4 topology into M2 with no exit criterion (critic's scope + false-error finding).
  - **Mitigation:** DROP the bw scaffold from M2 entirely. Model the socket as a FORWARD-ONLY Port<FwTransport>+Export<FwTransport> pair (fw OneOrMore). The bw Port/Export, crossed double-bind, and nb_transport_bw are deferred to M4. A test asserts a working rv32i LT socket does not error at complete_binding.
- **Risk:** On a bare Sim that never calls cx.module, flat-path Signal/Socket registration needs current_scope() but scope_stack is empty → panic (the implicit-root bootstrap was asserted but undesigned) (critic's breakage finding).
  - **Mitigation:** ObjectStore::new() seeds an IMPLICIT ROOT ObjectMeta and pushes it onto scope_stack, so current_scope() always returns root. Flat constructors register under root with no empty-scope panic. A test asserts ObjectStore::new yields a non-empty scope and insert() never panics.
- **Risk:** The elaboration hook returns Result but run_until returns () — a binding error has nowhere to surface on the flat path, and changing run_until's signature would break every call site (critic's unhandled-error finding).
  - **Mitigation:** Settle at the single hook call site in run_until: convert Err → systemrs_diag::report_fatal (a FATAL panic). run_until keeps its () signature; every existing call site is untouched. This is a clean upgrade over today's first-b_transport FATAL (errors now surface at elaboration, earlier and once).
- **Risk:** m.thread wraps add_thread whose body is FnOnce(&Ctx)+Send; a module body capturing !Send fields (Rc/RefCell) into m.thread won't compile — an ergonomics trap for the platform example (critic's Send-leak finding).
  - **Mitigation:** Document that module-scoped threads inherit the Send bound; design the platform example's cpu thread to capture only Copy handles (Copy socket/event ids). A trybuild compile-fail test documents that capturing an Rc-based field is rejected, so the constraint is explicit rather than a surprise.
- **Risk:** AttributeStore is a forward type reference: ObjectMeta declares attributes: Option<AttributeStore> in M2-02 but the type is 'defined' in M2-12 — M2-02 cannot compile standalone (critic's sequencing finding).
  - **Mitigation:** Define the minimal AttributeStore TYPE (the struct + field) in M2-02's attribute.rs so ObjectMeta compiles; M2-12 only fills in the get<T>/set<T> bodies and smoke test. M2-12 still depends_on M2-02; no cycle remains.
- **Risk:** Determinism: a HashMap name table could feed nondeterministic ordering into unique-name suffix assignment or callback order (§8 'no tie-break depends on HashMap order').
  - **Mitigation:** The HashMap names table is used ONLY for existence checks during uniquify, NEVER iterated to assign names or order callbacks. children Vec and the four per-bucket registries are append-ordered by insertion sequence (source order, deterministic). A determinism test asserts sibling inserts in a fixed source order always produce the same names/suffixes. Documented as an invariant on ObjectStore.
- **Risk:** Three overlapping guards for binding-after-start (compile-time typestate + runtime Phase-gate + eager build()) is more machinery than EC4 requires and multiplies drive-site coordination (critic's scope concern).
  - **Mitigation:** The minimal EC4 satisfier is the runtime Phase-gate Err (M2-07). The typestate (M2-10) adds the compile-error half the design (§6a, §14) calls for. The drive-site coordination is collapsed to a SINGLE source of truth — the Inner.elaborated latch — so build() drives eagerly and the run_until hook no-ops when the latch is set (no double-drive, no re-drive). If M2 is at risk, the typestate can slip and the runtime Err alone still satisfies EC4 (open question O-SCOPE).
- **Risk:** Reconciling the bespoke socket registry (consumed by memory.rs, rv32i SocketBus, both example binaries, three integration tests) to a Port+Export pair risks breaking the green rv32i LT path.
  - **Mitigation:** Keep the closure SocketRegistry as the resolved-interface STORAGE; layer the fw Port/Export and deferred bind OVER it; preserve bind/b_transport/register_b_transport/register_transport_dbg as signature-identical thin wrappers so call sites are untouched. Re-run rv32i as the gate. Deferred resolution surfacing an unbound-socket FATAL at complete_binding (vs first b_transport today) is a strict improvement; bound examples are unaffected.
